# Elixir 점진적 타입 마이그레이션 CLI 가이드

> Elixir v1.20의 점진적 타입 시스템을 기존 코드베이스에 안전하게 적용하는 단계별 CLI 코파일럿

## 🎯 문제 정의

### 현재 상황
Elixir v1.20에서 오랫동안 기다려온 점진적 타입 시스템이 공식 도입되었다. 수년간 `@spec`과 Dialyzer로 불완전한 타입 검사를 해오던 Elixir 개발자들에게 이것은 큰 변화다. 하지만 기존의 수천 줄짜리 Elixir 코드베이스에 새 타입 어노테이션을 어디서부터 어떻게 붙이기 시작해야 할지, 무엇이 타입 오류를 유발하는지 파악하는 것은 매우 혼란스럽다.

### 주요 고통 포인트
- **진입점 불명확**: 어느 모듈부터 타입을 붙여야 타입 혜택이 가장 클지 알 수 없다
- **영향 범위 추정 불가**: 특정 함수에 타입을 추가했을 때 호출 체인 전체에 어떤 파급이 생기는지 사전 파악이 어렵다
- **`@spec` vs 새 타입 시스템 혼재**: 기존 Dialyzer 스펙과 v1.20 타입 어노테이션의 충돌 지점을 수동으로 찾아야 한다
- **점진적 적용 전략 부재**: 전체를 한 번에 타입화하면 대규모 변경이 필요하고, 부분적으로 하면 타입 경계가 지저분해진다

### 영향받는 사용자
기존 Elixir 프로젝트(Phoenix 앱, 백엔드 서비스, 라이브러리)를 유지보수하는 개발자. 특히 v1.20으로 업그레이드를 앞두고 "타입을 도입해야 할까?"를 고민하는 팀 리드와 시니어 엔지니어.

## 💡 솔루션

### 핵심 개념
`ex_type_guide`는 Elixir 프로젝트를 정적 분석하여 **점진적 타입 마이그레이션 로드맵**을 생성하는 CLI 도구다. 코드베이스를 스캔해서 "어느 모듈을 먼저 타입화하면 최대 효과를 얻는지" 우선순위를 매기고, 각 함수에 붙일 타입 어노테이션 초안을 제안하며, 기존 `@spec`과의 충돌 여부를 미리 경고한다. 프론트엔드 프레임워크 없이 순수 Node.js CLI로 동작하며, 분석 결과는 Markdown 리포트 또는 대화형 터미널 UI로 출력된다.

### 동작 방식
1. **스캔**: `ex_type_guide scan ./lib` — 프로젝트 디렉토리의 `.ex` 파일을 파싱해 함수 호출 그래프와 `@spec` 현황을 수집한다
2. **우선순위 분석**: 호출 빈도, 외부 API 경계(컨트롤러/GenServer 콜백), `@spec` 누락 비율을 기반으로 타입화 우선순위 점수를 계산한다
3. **어노테이션 초안 생성**: 함수 시그니처와 기존 `@spec`을 기반으로 v1.20 타입 문법의 초안 어노테이션을 생성한다
4. **충돌 검증**: 기존 Dialyzer 스펙과 새 타입 시스템 간 의미 충돌 가능 지점을 `[WARN]` 태그로 마킹한다
5. **리포트 출력**: Markdown 파일 또는 `--interactive` 플래그로 터미널 내 탐색 가능한 결과를 출력한다

### 핵심 기능
- **타입화 우선순위 맵**: 모듈별 점수(호출빈도 × 영향범위 × @spec 공백률)로 "여기부터 시작하세요" 가이드 제공
- **어노테이션 초안 제안**: 함수별로 `@type` 및 새 타입 어노테이션 코드 스니펫 자동 생성 (복사-붙여넣기 가능)
- **`@spec` 충돌 검사기**: Dialyzer 스펙과 v1.20 타입 사이 의미 불일치를 사전 경고
- **마이그레이션 진행 추적**: 이전 스캔 결과와 비교해 "이번 주에 X개 함수 타입화 완료" 같은 진행 상황 표시
- **대화형 터미널 UI**: `inquirer.js` 기반으로 모듈을 탐색하며 제안된 어노테이션을 수락/거절 가능

## 🏗️ 기술 아키텍처

### 시스템 개요

```
┌───────────────────────────┐
│   CLI Entry (Commander.js) │
│   ex_type_guide scan/report│
└──────────┬────────────────┘
           │
┌──────────▼────────────────┐
│   .ex 파일 파서            │
│   (regex + 간단한 AST      │
│    패턴 매칭으로 함수/스펙  │
│    추출 — elixir AST 호출 │
│    없이 휴리스틱 분석)      │
└──────────┬────────────────┘
           │
┌──────────▼────────────────┐
│   함수 호출 그래프 빌더     │
│   (모듈 간 의존 관계 맵)    │
└──────────┬────────────────┘
           │
┌──────────▼────────────────┐
│   우선순위 스코어러         │
│   (call_freq × impact      │
│    × spec_gap_rate)        │
└──────────┬────────────────┘
           │
┌──────────▼────────────────┐
│   어노테이션 초안 생성기    │
│   (템플릿 기반 코드 생성)   │
└──────────┬────────────────┘
           │
    ┌──────┴──────┐
    ▼             ▼
┌─────────┐  ┌──────────────┐
│Markdown │  │ 대화형 터미널 │
│ 리포트  │  │ UI (inquirer)│
└─────────┘  └──────────────┘
```

### 기술 스택 선택 이유
- **Node.js + TypeScript**: 프론트엔드 프레임워크 없이 순수 CLI 도구로 제작. 주간 제약 조건 충족. npm으로 `npx ex_type_guide` 즉시 실행 가능
- **Commander.js**: CLI 인자 파싱의 표준 라이브러리. 학습 비용 최소화
- **inquirer.js**: 터미널 내 대화형 UI 구현. 사용자가 제안된 어노테이션을 수락/수정할 수 있는 REPL-like 경험
- **Elixir 파일 파싱**: 완전한 AST 파서 대신 정규표현식 + 패턴 매칭으로 `def`, `defp`, `@spec`, `@moduledoc` 추출. 외부 Elixir 런타임 의존성 없이 동작
- **로컬 JSON 캐시**: 이전 스캔 결과를 `.ex_type_guide_cache.json`에 저장해 진행 상황 추적

### 주요 기술적 도전
1. **Elixir 문법 파싱 (프레임워크 없이)**: 완전한 Elixir AST 파서를 Node.js에서 구현하는 대신, `def/defp` 시그니처와 `@spec` 패턴을 정규표현식으로 추출하는 휴리스틱 접근. 99%의 실용적 케이스를 커버하면서 구현 복잡도를 대폭 줄인다
2. **모듈 간 호출 그래프 구성**: 정적 분석만으로 Elixir의 동적 디스패치(`:apply`, 메타프로그래밍)까지 추적하기는 불가능. "알 수 없는 호출"을 명시적으로 표기하는 불확실성 모델 설계 필요

## 🎨 무엇이 흥미로운가?

### 독특한 각도
대부분의 Elixir 도구는 Elixir 런타임 위에서 동작한다. 이 도구는 **TypeScript CLI로 Elixir 코드를 분석**한다는 역발상을 취한다. Elixir를 설치하지 않은 CI 환경에서도 실행 가능하고, npm으로 배포할 수 있다. 또한 타입 시스템 도입 자체를 돕는 도구는 언어별로 드물다 — 대부분은 이미 타입이 있다고 가정하고 검사한다.

### 혁신 포인트
- Elixir v1.20이라는 **갓 출시된 언어 기능**을 타겟으로 한 도구 (남들보다 먼저)
- "타입을 어떻게 쓰는가"가 아니라 **"어디서부터 타입을 도입하는가"**라는 메타 질문에 답하는 도구
- 프론트엔드 프레임워크 없이 TypeScript로만 구현 — 주간 제약 조건을 자연스러운 설계 결정으로 변환

### 학습 기회
- Elixir 문법 구조와 `@spec` / 새 타입 시스템의 차이점 심층 이해
- 정적 분석 없이 언어를 파싱하는 휴리스틱 전략 설계
- Commander.js + inquirer.js로 전문적인 CLI UX 구현
- 그래프 알고리즘 (호출 그래프, 우선순위 스코어링)

## 📊 시장 & 검증

### 유사 프로젝트
- **Dialyzer** (Erlang 생태계): 타입 추론 검사기. 이 도구는 검사가 아니라 **마이그레이션 가이드**를 제공
- **TypeScript Migration Guide Tools** (ts-migrate, etc.): JavaScript→TypeScript 마이그레이션 도구에서 컨셉 차용. Elixir 생태계에는 이런 도구가 없음
- **credo** (Elixir 린터): 코드 스타일 검사. 타입 마이그레이션과는 전혀 다른 목적

### 왜 이 틈새가 존재하는가
Elixir v1.20의 점진적 타입 시스템은 2026년에 막 도입된 기능이다. 생태계가 따라잡기 전의 시간 창(window)이 존재한다. `ts-migrate`가 TypeScript 초창기에 했던 역할을 Elixir 타입 생태계에서 할 수 있다.

### 성공 지표
- [ ] Elixir 프로젝트 5개 이상에서 올바른 우선순위 리포트 생성
- [ ] 제안된 어노테이션이 실제 v1.20 문법과 90% 이상 일치
- [ ] HN/ElixirForum에서 커뮤니티 피드백 10개 이상 수집

## 🚀 구현 로드맵

### Phase 1: MVP (1주차)
**목표**: 기본 스캔 + Markdown 리포트 출력
- `.ex` 파일에서 `def/defp` 함수 시그니처와 `@spec` 추출
- 모듈별 `@spec` 커버리지 계산
- 우선순위 점수 기반 상위 20개 함수 리스트 출력
- `ex_type_guide scan ./lib --output report.md` 동작

### Phase 2: 어노테이션 제안 (2주차 전반)
**목표**: 함수별 타입 어노테이션 초안 자동 생성
- `@spec` 기반으로 v1.20 타입 문법 초안 변환
- `@spec` 없는 함수는 패턴 매칭 분기로 유니온 타입 추론
- 기존 `@spec`과 의미 충돌 가능 지점 `[WARN]` 마킹

### Phase 3: 대화형 UI + 진행 추적 (2주차 후반)
**목표**: 실제로 사용하고 싶은 도구
- `--interactive` 플래그로 모듈별 탐색 + 어노테이션 수락/편집
- `.ex_type_guide_cache.json`으로 이전 스캔과 비교한 진행률 표시
- `npx ex_type_guide` 즉시 실행 가능하도록 npm 패키지 배포

### 기술 마일스톤
| 단계 | 산출물 | 성공 기준 |
|------|--------|----------|
| 1주차 | 스캔 + Markdown 리포트 | Phoenix 프로젝트에서 올바른 우선순위 리스트 출력 |
| 2주차 초 | 어노테이션 초안 생성 | 제안된 어노테이션의 90%가 v1.20 문법에 맞음 |
| 2주차 말 | 대화형 UI + npm 배포 | `npx ex_type_guide scan` 즉시 실행 가능 |

## 🌱 배우게 될 것들

- **Elixir 타입 시스템 심층 이해**: v1.20의 점진적 타입이 Dialyzer `@spec`과 어떻게 다른지 직접 분석하며 이해

- **전문적 CLI UX**: Commander.js + inquirer.js로 개발자 도구 수준의 터미널 경험 구현

### 성장 기회
- "새 언어 기능이 릴리즈되자마자 도구를 만든다"는 시장 선점 사고방식 실습
- TypeScript로 타 언어의 소스 코드를 분석하는 크로스-언어 정적 분석 경험
- npm 패키지 배포 및 `npx` 실행 환경 최적화

### 도전 구간
`@spec`의 Dialyzer 타입과 v1.20 타입 어노테이션 문법 사이의 의미론적 차이를 올바르게 매핑하는 것 — Elixir 언어 스펙을 깊이 읽어야 하는 구간이다.
