# 고고학 유물 분류 기반 버그 트리아지 게임

> 쌓인 GitHub 이슈들을 발굴 현장처럼 탐험하며 분류하고, XP와 뱃지를 획득하는 개발자용 게임화 트리아지 도구

## 🎯 문제

### 현재 상황

오픈소스 프로젝트나 팀 레포지토리에는 수백 개의 미분류 이슈가 쌓인다. 라벨이 없고, 우선순위도 없고, 중복 이슈도 섞여 있다. 이 이슈들을 정리하는 트리아지 작업은 중요하지만, 단조롭고 보람이 없어서 아무도 자발적으로 하지 않는다.

### 핵심 고통 포인트

- **보상 없는 노동**: 이슈 100개를 분류해도 아무것도 남지 않는다. 커밋 기록에도 안 남고, 동료도 알아채지 못한다.
- **압도감**: 이슈 목록 페이지를 열면 숫자에 압도되어 어디서부터 시작해야 할지 모른다.
- **지루함**: 같은 UI, 같은 반복 동작. 오래 집중하기 어렵다.
- **컨텍스트 손실**: 분류하다 중간에 그만두면 어디까지 했는지 알기 어렵다.

### 누가 겪는가?

오픈소스 메인테이너, 이슈 정리에 기여하고 싶은 컨트리뷰터, 백로그가 쌓인 팀의 프런트엔드/풀스택 개발자.

## 💡 해결책

### 핵심 콘셉트

이슈 트리아지를 **고고학 발굴**로 재해석한다. 각 이슈는 땅 속에 묻힌 유물이다. 사용자는 발굴사가 되어 레이어(날짜순 지층)를 파고들며 유물을 꺼내 분류한다. 분류 행동 하나하나가 경험치(XP)로 기록되고, 레벨이 오르며, 희귀 뱃지를 획득한다. GitHub API를 통해 실제 레포지토리와 연동되므로 게임 결과가 곧 실제 이슈 관리로 직결된다.

### 작동 방식

1. **발굴지 선택**: GitHub 레포지토리 URL을 입력하면 미분류 이슈들이 "지층(Layer)"으로 구성된다. 최근 이슈는 얕은 층, 오래된 이슈는 깊은 층에 있다.
2. **유물 발굴**: 한 번에 이슈 하나씩 카드 형태로 등장한다. 배경에는 발굴 현장 UI(흙 레이어, 브러시 애니메이션)가 있다.
3. **분류 판정**: 사용자는 이슈를 `버그 / 기능 요청 / 중복 / 문서 / 유효하지 않음` 중 하나로 분류하고 우선순위(Critical / High / Low)를 태깅한다. 결정하면 실제 GitHub 라벨이 적용된다.
4. **보상 시스템**: 분류 속도, 연속 분류 스트릭, 정확도(다른 사용자와 일치율)에 따라 XP와 뱃지 획득.
5. **발굴 지도**: 세션이 끝나면 "오늘의 발굴 성과" 다이어그램이 생성된다 — 몇 개의 유물을 찾았고, 어느 지층까지 팠는지 시각화.

### 핵심 기능

- **지층 기반 진행도**: 이슈를 날짜별 지층으로 묶어 전체 레포지토리의 "발굴 완료율"을 직관적으로 파악 가능
- **XP & 레벨 시스템**: 세션 간 누적 경험치, 레벨업 알림, 리더보드(선택)
- **희귀 유물 이벤트**: 오래된 이슈나 많은 반응을 받은 이슈가 "희귀 유물"로 등장해 보너스 XP 제공
- **연속 분류 스트릭**: 3일 연속 분류 시 "현장 감독" 뱃지 등 특별 보상
- **발굴 일지**: 분류 히스토리를 고고학 발굴 보고서 형태로 마크다운 export

### 시스템 개요

```
┌─────────────────────────────────┐
│         Svelte Frontend         │
│  ┌──────────┐  ┌─────────────┐  │
│  │ Game UI  │  │ Progress    │  │

│  └────┬─────┘  └──────┬──────┘  │
└───────┼───────────────┼─────────┘
        │               │
┌───────▼───────────────▼─────────┐
│         State Layer             │
│  XP Engine / Badge System /     │
│  Session Tracker                │
└───────────────┬─────────────────┘
                │
┌───────────────▼─────────────────┐
│         GitHub API              │
│  Issues 조회 / Label 적용       │
│  (OAuth 인증)                   │
└─────────────────────────────────┘
```

### 기술 스택 선택 근거

- **Svelte**: 주간 제약(No React) 준수. 상태 관리가 단순하고, 애니메이션 트랜지션이 내장되어 발굴 UI 표현에 적합
- **GitHub API (REST v3)**: 이슈 조회 및 라벨 적용에 충분. GraphQL보다 진입 장벽 낮음
- **LocalStorage + IndexedDB**: XP/레벨/뱃지 데이터 로컬 저장. 서버 없이 동작
- **CSS Animations**: 브러시질 효과, 카드 발굴 애니메이션을 순수 CSS로 구현

### 주요 기술적 도전

1. **GitHub API Rate Limiting**: 인증 없이는 60 req/hour. OAuth 앱 등록으로 5000 req/hour 확보 필요. 이슈를 세션 시작 시 일괄 fetch하여 캐싱.
2. **게임 밸런스 설계**: XP 곡선이 너무 빠르면 보람 없고, 너무 느리면 지친다. 피보나치 수열 기반 레벨업 임계값 설계 검토.

### 독창적인 각도

버그 트리아지와 고고학 발굴의 메타포는 구조적으로 잘 맞는다. 두 작업 모두 "묻혀 있는 것을 꺼내 분류하고 기록하는" 작업이다. 이 유사성을 UI로 직접 표현하는 개발자 도구는 존재하지 않는다. 기존 게임화 이슈 트래커들은 포인트 시스템만 붙인 수준이지만, 이 프로젝트는 발굴이라는 내러티브를 UI 전체에 일관되게 반영한다.

### 혁신 포인트

- **공간적 메타포**: 이슈 목록을 2D 리스트가 아닌 "파고들어가는 지층"으로 표현
- **고고학 보고서 export**: 분류 결과를 발굴 일지 형식으로 출력 — 커밋 메시지보다 더 재미있는 기록
- **게임 메카닉이 실제 작업과 1:1 대응**: 게임에서의 행동이 곧 GitHub 레포지토리 상태를 변경

### 학습 기회

- Svelte의 트랜지션/애니메이션 시스템 (React와 다른 접근법)
- GitHub OAuth App 구현 (PKCE flow)
- 게임 밸런스 설계의 수학적 원리 (XP 곡선 설계)
- IndexedDB를 이용한 오프라인 퍼스트 상태 관리

## 📊 시장 및 검증

### 유사 프로젝트

- **GitHub Issues 기본 UI**: 트리아지 기능은 있지만 동기부여 요소 없음
- **ZenHub / Linear**: 팀 이슈 관리 도구이지만 게임화 없음, 유료
- **First Timers Only**: 기여자 온보딩 도구. 트리아지 게임화 없음

### 이 공백이 존재하는 이유

트리아지는 "관리 작업"으로 분류되어 개발자 경험 설계에서 소외되어 왔다. 게임화는 학습 앱(Duolingo 등)에 많이 쓰이지만 개발자 도구에는 드물다. Octocat 뱃지 정도를 제외하면 GitHub 생태계의 게임화는 미개발 영역이다.

### 성공 지표

- **Svelte 트랜지션 시스템**: React의 `useEffect` + CSS 애니메이션 조합 없이 선언적으로 UI 전환 처리하는 방법

- **게임 밸런스 수학**: XP 곡선, 레벨업 임계값, 보상 빈도가 동기부여에 미치는 영향
- **IndexedDB 패턴**: LocalStorage보다 복잡한 구조화 데이터를 클라이언트에서 관리하는 방법

### 성장 기회

- 비게임 도메인에 게임 메카닉을 설계할 때의 밸런스 고려법을 내면화

- API Rate Limit을 고려한 캐싱 전략 설계

### 도전 구간

Svelte는 기존에 깊게 다루지 않은 프레임워크다. 특히 Svelte의 스토어 시스템과 React의 상태 관리 철학 차이를 실전에서 경험하는 것이 주요 학습 포인트다.
