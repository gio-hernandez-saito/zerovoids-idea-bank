# 레시피 의존성 그래프 — 요리를 코드처럼 리팩터링하기

> 요리 레시피를 코드 의존성 그래프로 파싱해, 병렬 조리 가능한 단계를 자동으로 찾아주는 워크플로우 최적화 도구

## 🎯 문제

### 현재 상황
대부분의 레시피는 단순한 순서 목록으로 작성된다. "1번을 한 후 2번, 그 다음 3번"처럼. 하지만 실제로 많은 조리 단계는 독립적이거나 병렬로 진행할 수 있다. 이 구조를 사람이 눈으로 읽어 파악하는 건 번거롭고, 특히 복잡한 요리일수록 최적 타이밍을 놓치기 쉽다.

### 고통 포인트
- **비효율적인 직렬 조리**: 오븐 예열과 채소 손질은 동시에 할 수 있는데 순서대로 나열되어 시간 낭비가 발생
- **재료 공유 파악 어려움**: 여러 요리를 동시에 할 때 어떤 재료가 겹치는지 한눈에 보기 어려움
- **타이밍 실수**: 끓는 물이 준비되기 전에 면을 넣거나, 소스가 식기 전에 접시에 담지 못하는 타이밍 오류

### 누가 이 문제를 겪는가?
집에서 복잡한 요리를 즐기는 홈쿡, 여러 코스 요리를 준비하는 아마추어 셰프, 그리고 요리를 시스템처럼 접근하고 싶은 개발자 성향의 사람들.

## 💡 솔루션

### 핵심 개념
레시피를 입력받아 각 조리 단계를 노드로, 단계 간 의존 관계(A가 끝나야 B를 시작할 수 있음)를 엣지로 표현하는 DAG(방향 비순환 그래프)를 생성한다. 코드 빌드 시스템의 작업 스케줄러처럼, 레시피를 "병렬 실행 가능한 태스크 그래프"로 변환해 최적 조리 순서와 예상 총 소요 시간을 계산한다.

### 작동 방식
1. **레시피 입력**: 자연어 또는 구조화된 JSON으로 레시피와 각 단계의 소요 시간을 입력
2. **의존성 분석**: 각 단계가 어떤 재료/이전 단계 결과물에 의존하는지 파싱
3. **DAG 시각화**: D3.js로 의존성 그래프를 렌더링 — 병렬 가능 단계는 같은 층에 표시
4. **최적 스케줄 계산**: 임계 경로(Critical Path) 알고리즘으로 최소 총 조리 시간 계산
5. **인터랙티브 타임라인**: 간트 차트 스타일로 "언제 무엇을 시작할지" 타임라인 출력

### 주요 기능
- **자동 의존성 추출**: "소스가 완성된 후 면을 넣는다" 같은 자연어에서 의존 관계 파싱
- **병렬 실행 최적화**: 독립적인 단계들을 묶어 병렬 조리 플랜 생성
- **멀티 레시피 모드**: 여러 요리를 동시에 진행할 때의 통합 타임라인 계산
- **재료 공유 탐지**: 같은 재료를 사용하는 단계 간 충돌/최적화 포인트 하이라이트

## 🏗️ 기술 아키텍처

### 시스템 구조

```
┌─────────────────────────┐
│     레시피 파서          │  (자연어 → 구조화 데이터)
│  (regex + rule-based)   │
└──────────┬──────────────┘
           │
┌──────────▼──────────────┐
│     DAG 빌더            │  (의존성 그래프 생성)
│  (topological sort)     │
└──────────┬──────────────┘
           │
┌──────────▼──────────────┐
│   임계 경로 계산기       │  (Critical Path Method)
└──────────┬──────────────┘
           │
    ┌──────┴──────┐
    ▼             ▼
┌───────┐    ┌────────────┐
│ DAG   │    │  타임라인  │
│ 시각화│    │  간트 차트 │
│ (D3)  │    │  (D3)      │
└───────┘    └────────────┘
```

### 기술 스택 근거
- **TypeScript**: 그래프 노드/엣지 타입을 명확하게 모델링하기 위해 필수
- **D3.js**: DAG 레이아웃(dagre 연동)과 간트 차트 커스터마이징에 탁월
- **React**: 레시피 입력 폼과 시각화 패널을 컴포넌트로 분리
- **dagre 또는 d3-dag**: 계층적 그래프 레이아웃 알고리즘 제공

### 핵심 기술 도전
1. **자연어 의존성 파싱**: "A가 완성된 후", "동시에", "미리 준비해둔" 같은 한국어/영어 패턴 인식. 초기엔 rule-based로 시작하고 점진적으로 정교화
2. **DAG 레이아웃 품질**: 노드 수가 늘어날수록 엣지 교차 최소화가 어려움. dagre 라이브러리로 Sugiyama 알고리즘 활용
3. **임계 경로와 여유 시간 계산**: 각 노드의 EST/LST(Earliest/Latest Start Time) 계산로직 구현

## 🎨 왜 이게 흥미로운가?

### 독특한 관점
소프트웨어 빌드 시스템(Makefile, Gradle, Nx)이 해결하는 문제 — "태스크 의존성 기반 병렬 실행 최적화" — 를 요리 레시피에 그대로 적용한다. 요리책 어디에도 이런 관점이 없다. 레시피를 "코드"로 보는 개발자의 시각이 만들어내는 새로운 도구.

### 혁신 포인트
- 임계 경로 방법론(CPM)을 요리에 적용한 첫 번째 인터랙티브 도구
- 레시피를 선형 리스트가 아닌 DAG로 표현하는 새로운 메타포
- 개발자적 사고(의존성 관리, 병렬화)를 비개발자 도메인에 이식

### 학습 기회
- 그래프 알고리즘 (DAG, 위상 정렬, 임계 경로)
- D3.js 고급 레이아웃 (계층형 그래프 렌더링)
- 자연어 파싱의 기초 (rule-based NLP)

## 📊 시장 및 검증

### 유사 프로젝트
- **Paprika, Whisk 등 레시피 앱**: 레시피 저장/검색에 집중, 워크플로우 최적화 없음
- **프로젝트 관리 도구 (Gantt)**: 요리 도메인에 특화되지 않음
- **Cooklang**: 레시피 마크업 언어이지만 의존성 시각화나 스케줄 최적화 없음

### 이 공백이 존재하는 이유
요리 앱 시장은 레시피 콘텐츠에 집중했고, 조리 프로세스 최적화는 전문 셰프의 암묵지로만 존재했다. 개발자 도구의 의존성 관리 개념을 요리에 접목한 시도는 거의 없다.

### 성공 지표
- [ ] 10단계 이상의 복잡한 레시피를 올바르게 DAG로 변환
- [ ] 임계 경로 계산 결과가 실제 조리 시간 단축으로 이어짐을 검증
- [ ] 개발자 커뮤니티(HN, Reddit r/programming)에서 공유 후 반응 확인
