# 고고학 발굴 레이어 기반 코드 변경 이력 탐험기

> 코드베이스를 고고학 유적지처럼 발굴하라 — 커밋 이력의 지층을 파헤치며 코드가 어떻게 쌓여왔는지 탐험한다.

## 🎯 문제

### 현재 상황

`git log`, `git blame` 같은 전통적인 Git 도구들은 코드 변경 이력을 시간순 목록이나 단순 줄별 주석으로 보여준다. 이 방식은 "누가 언제 바꿨는지"는 알 수 있지만, 코드가 어떤 맥락과 흐름 속에서 쌓여왔는지의 **층위적 구조**를 파악하기 어렵다. 고고학자가 유적지의 지층(stratigraphy)을 분석해 시대별 문명의 흔적을 발굴하듯, 코드베이스에도 기능 추가 → 리팩터링 → 버그 수정 → 최적화라는 명확한 지층이 존재한다.

### 주요 불편함

- **맥락 없는 이력**: `git log --all`은 수백 개의 커밋 해시를 나열하지만, 어느 "시대"에 어떤 흐름이 있었는지 감이 없음
- **blame의 한계**: `git blame`은 줄별 최종 수정자를 보여주지만, 그 줄이 어떤 역사적 레이어에서 왔는지 보여주지 않음
- **고통스러운 이진 탐색**: 버그 도입 시점을 찾으려면 `git bisect`를 직접 수동으로 조작해야 하고, 결과 해석도 어려움
- **"왜" 없는 "무엇"**: 커밋 메시지는 있지만, 연관된 변경들이 어떤 고고학적 "발굴 층위"를 이루는지 가시화가 없음

### 누가 이 불편함을 겪는가?

오래된 레거시 코드베이스를 인수받은 개발자, 1년 이상 된 프로젝트에서 버그 원인을 추적하는 개발자, 코드 리뷰 중 특정 로직의 진화 흐름을 이해해야 하는 시니어 개발자들이 주로 이 문제를 겪는다.

## 💡 해결책

### 핵심 개념

**Stratum** (스트라텀)은 Git 저장소를 고고학 발굴 현장처럼 시각화하는 CLI 도구다. 커밋 이력을 시간 축 기준으로 "지층"으로 분류하고, 각 파일이나 함수가 어느 지층에서 처음 등장했는지, 어떤 지층에서 크게 변형되었는지를 인터랙티브하게 탐험할 수 있게 한다. 프론트엔드 프레임워크 없이 순수 Node.js CLI (`blessed` TUI 라이브러리)로 구현한다.

### 동작 방식

1. **발굴 시작**: 저장소 경로를 지정하면, Git 이력을 분석해 자동으로 "시대 구분"을 제안한다 (대규모 구조 변경, 태그, 브랜치 병합 패턴 기반)
2. **지층 탐험**: TUI 화면에서 시간축을 따라 레이어가 쌓인 단면도를 보여준다. 각 레이어는 해당 기간의 변경 밀도와 종류(기능 추가 / 리팩터링 / 버그 수정)를 색상으로 구분한다
3. **유물 발굴**: 특정 레이어를 선택하면 해당 시대에 등장하거나 크게 변형된 파일/함수 목록 — 즉 "유물" — 이 표시된다
4. **DNA 추적**: 특정 코드 조각(함수명 또는 줄 범위)을 지정하면, 그 코드가 어느 레이어에서 기원해 어떻게 변이했는지 계보(lineage)를 추적한다
5. **발굴 보고서**: JSON/Markdown 형식으로 탐험 결과를 저장하여 팀과 공유한다

### 핵심 기능

- **자동 시대 구분 (Auto-Stratification)**: 커밋 패턴, 파일 변경 규모, 태그를 분석해 "청동기 시대 → 철기 시대" 같은 코드 발전 단계를 자동으로 식별
- **지층 단면도 TUI**: `blessed` 기반의 터미널 UI로 시간 축 + 변경 밀도 히트맵을 시각화. 프레임워크 없이 순수 CLI
- **코드 계보 추적 (Lineage Tracer)**: 특정 함수나 심볼의 전체 생애를 추적 — 탄생, 변형, 이동, 소멸
- **발굴 현장 노트**: 각 레이어에 팀 주석(annotation)을 남길 수 있는 `.stratum.json` 파일 생성

## 🏗️ 기술 아키텍처

### 시스템 구조

```
┌─────────────────────────────────────────────┐
│              CLI Entry (stratum)             │
│            commander.js 기반 명령 파싱        │
└──────────────────┬──────────────────────────┘
                   │
         ┌─────────▼─────────┐
         │   Git Adapter     │
         │   (simple-git)    │
         │  커밋/diff/blame   │
         │  데이터 수집       │
         └─────────┬─────────┘
                   │
         ┌─────────▼─────────┐
         │  Stratifier       │
         │  시대 구분 엔진    │
         │  - 커밋 클러스터링 │
         │  - 변경 분류       │
         │  - 계보 추적       │
         └─────────┬─────────┘
                   │
         ┌─────────▼─────────┐
         │   TUI Renderer    │
         │   (blessed)       │
         │  단면도 / 유물 뷰  │
         │  키보드 네비게이션 │
         └─────────┬─────────┘
                   │
         ┌─────────▼─────────┐
         │  Report Exporter  │
         │  JSON / Markdown  │
         └───────────────────┘
```

### 기술 스택 선택 이유

- **Node.js + TypeScript**: Git 저장소 분석과 파일 I/O에 최적. 주간 제약(프론트엔드 프레임워크 금지)에 부합
- **simple-git**: Git 명령을 Promise 기반으로 래핑하는 성숙한 라이브러리. 커밋 이력, diff, blame 데이터 추출에 사용
- **blessed**: Node.js용 ncurses 스타일 TUI 라이브러리. 브라우저 없이 터미널에서 레이아웃과 색상 제어 가능
- **chalk**: 터미널 색상 출력으로 레이어 유형(기능/버그/리팩터링)을 시각적으로 구분
- **commander.js**: CLI 서브커맨드 파싱 (`stratum dig`, `stratum trace`, `stratum report`)

### 핵심 기술 도전 과제

1. **자동 시대 구분 알고리즘**: 커밋 이력에서 의미 있는 "시대 경계"를 자동으로 찾는 것이 가장 어려운 부분. 커밋 메시지 패턴 분석, 변경 파일 수의 급격한 증감, 태그/브랜치 병합 포인트를 복합적으로 활용하는 휴리스틱 클러스터링 접근법을 사용할 계획
2. **대용량 저장소 성능**: 수만 개의 커밋을 가진 저장소에서 전체 이력을 메모리에 올리는 것은 비현실적. 증분 로딩(lazy loading)과 커밋 샘플링 전략이 필요

## 🎨 흥미로운 점

### 독특한 접근 방식

기존 Git 시각화 도구들(`gitk`, `tig`, `lazygit`)이 모두 "커밋 목록" 또는 "브랜치 그래프" 패러다임을 사용하는 것과 달리, Stratum은 **시간을 지질학적 층위**로 재해석한다. 이 은유는 단순한 UI 트릭이 아니라 — "어느 시대에 이 코드가 등장했는가?"라는 질문 방식 자체를 바꿔, 개발자가 코드를 고고학적 시각으로 이해하도록 돕는다.

### 혁신 포인트

- **시간을 공간으로 매핑**: 커밋 이력의 시간 흐름을 물리적 레이어 단면도로 변환하는 발상
- **의도 분류 자동화**: 커밋 메시지와 변경 패턴을 분석해 "기능 추가 / 버그 수정 / 리팩터링" 레이어를 자동 색상 구분
- **코드 계보 개념**: 특정 코드 조각이 어느 레이어에서 기원했는지 추적하는 DNA 분석 은유
- **프레임워크 없는 순수 CLI**: 브라우저나 프론트엔드 없이 터미널만으로 풍부한 시각화 경험

### 학습 기회

- `blessed`/`ncurses` 스타일 TUI 구축 — 터미널 레이아웃 엔진의 내부 동작 이해
- `simple-git`으로 Git plumbing 명령 조합 및 대용량 이력 처리
- 시계열 클러스터링 휴리스틱 설계 경험
- Node.js 스트리밍 I/O 패턴 (대용량 저장소 대응)

## 📊 시장 검증

### 유사 프로젝트

- **tig** (github.com/jonas/tig): Git용 ncurses TUI. 커밋 목록/브랜치 그래프 시각화이지만 지층 개념 없음
- **git-sizer** (github.com/github/git-sizer): 저장소 크기 분석 CLI. 시간적 레이어 탐험 없음
- **lazygit** (github.com/jesseduffield/lazygit): 풍부한 Git TUI이지만 이력 "발굴" 관점이 아닌 조작 도구
- **git-of-theseus** (github.com/erikbern/git-of-theseus): 코드 생존율 분석 Python 스크립트. 시각화 없고 탐험형 인터페이스 없음

### 이 공백이 존재하는 이유

기존 Git 도구들은 대부분 "현재 상태 조작" 또는 "이력 검색"에 초점을 맞춰 설계되었다. "코드가 어떻게 진화해왔는가"를 탐험적으로 이해하는 도구는 거의 없으며, 특히 고고학적 층위 은유를 적용한 CLI 도구는 전무하다.

## 🌱 학습 성과

### 새로운 기술과 개념

- **`blessed` TUI 엔진**: ncurses 스타일 레이아웃, 이벤트 루프, 터미널 색상 제어 — 브라우저 없이 풍부한 UI를 만드는 완전히 다른 패러다임
- **Git 내부 구조 심화**: plumbing 명령 (`git log --follow`, `git diff --word-diff`, `git log -S`) 조합으로 코드 이력 데이터 추출
- **시계열 클러스터링**: 정해진 기준 없이 커밋 흐름에서 의미 있는 경계를 찾는 휴리스틱 설계 경험
- **Node.js 스트리밍 패턴**: 대용량 Git 이력을 메모리 효율적으로 처리하는 스트리밍 파이프라인

### 성장 기회

- 기존에 React/D3 기반 시각화에 익숙한 사람이 "순수 CLI 시각화"라는 완전히 다른 렌더링 패러다임을 경험
- 추상적인 "코드 이력"을 구조화된 데이터 모델로 설계하는 도메인 모델링 능력 향상
- 휴리스틱 알고리즘의 설계와 검증 — 정답이 없는 문제를 실용적으로 해결하는 사고법

### 도전 지점

터미널 TUI는 DOM과 완전히 다른 레이아웃 모델을 사용하며, 이벤트 처리와 리렌더링 방식이 낯설다. `blessed`의 박스 모델과 키 이벤트 처리를 처음부터 익혀야 하는 가파른 학습 곡선이 있다.
