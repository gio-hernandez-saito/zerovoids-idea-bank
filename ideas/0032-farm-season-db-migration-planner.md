# 농업 스케줄 기반 데이터베이스 마이그레이션 플래너

> 마이그레이션을 작물 재배 일정처럼 관리하라 — 씨앗(스키마 변경)이 언제 심어지고, 언제 수확(배포)되는지 한눈에.

## 🎯 문제 정의

### 현재 상황

대규모 프로젝트에서 데이터베이스 마이그레이션은 수십 개의 파일이 서로 의존하며 복잡하게 얽혀 있다. 기존 마이그레이션 도구(Flyway, Liquibase, Prisma Migrate 등)는 파일을 순서대로 실행하는 것엔 능숙하지만, **"이 마이그레이션들을 언제 어떤 환경에 배포할 수 있는가"** 라는 일정 계획과 의존 관계 시각화에는 취약하다.

### 주요 고통 포인트

- **배포 충돌**: 두 팀이 같은 테이블을 동시에 수정하는 마이그레이션을 만들어 병합 시 충돌 발생
- **의존 관계 불명확**: `migration_003`이 `migration_007` 이후에 실행되어야 한다는 사실이 코드 어딘가에 묻혀 있음
- **배포 윈도우 낭비**: "이 마이그레이션은 트래픽이 낮은 새벽에만 실행해야 한다"는 맥락이 문서화되지 않음
- **롤백 경로 불투명**: 어떤 마이그레이션을 되돌릴 때 함께 롤백해야 하는 것들이 명확하지 않음

### 누가 겪는 문제인가?

마이그레이션 파일이 30개를 넘어서기 시작한 팀의 백엔드/풀스택 개발자, 그리고 여러 마이크로서비스 DB를 동시에 관리하는 DevOps 엔지니어.

## 💡 해결책

### 핵심 개념

**HarvestPlan**은 농업의 작물 재배 달력(crop calendar)에서 영감을 받은 CLI 기반 마이그레이션 계획 도구다. 농부가 "이 씨앗은 서리가 끝난 뒤 심고, 저 씨앗은 앞 작물 수확 후에 심는다"고 관리하듯, 개발자는 마이그레이션 간의 의존 관계와 실행 조건(환경, 시간 윈도우, 선행 마이그레이션)을 선언적으로 정의한다. 도구는 이를 분석해 **안전한 배포 순서**와 **병렬 실행 가능한 묶음**을 제안한다.

### 작동 방식

1. **씨앗 심기 (Annotate)**: 각 마이그레이션 파일 상단에 간단한 주석 메타데이터 추가
   ```sql
   -- @depends: migration_005, migration_006
   -- @window: low-traffic
   -- @rollback-safe: true
   -- @env: production, staging
   ```
2. **성장 추적 (Analyze)**: `harvestplan analyze` 실행 → 의존 관계 DAG(방향 비순환 그래프) 자동 생성
3. **수확 계획 (Plan)**: `harvestplan plan --env=production --window=tonight` 실행 → 오늘 밤 실행 가능한 마이그레이션 묶음과 순서 출력
4. **수확 (Execute)**: 계획된 순서대로 실행, 실패 시 롤백 경로 자동 제안
5. **토양 리포트 (Report)**: 마이그레이션 이력과 스키마 "건강도" 요약 출력

### 핵심 기능

- **의존 관계 DAG 시각화**: 터미널에서 ASCII 트리 또는 Ink 기반 인터랙티브 그래프로 마이그레이션 의존 관계 표시
- **배포 윈도우 스케줄러**: `low-traffic`, `maintenance-window`, `anytime` 등 배포 조건 태깅 및 필터링
- **충돌 감지기**: 같은 테이블/컬럼을 수정하는 병렬 마이그레이션 브랜치 자동 감지 및 경고
- **롤백 체인 계산기**: 특정 마이그레이션 롤백 시 함께 되돌려야 하는 의존 마이그레이션 목록 자동 계산
- **DB 무관 파서**: Flyway, Prisma, 커스텀 SQL 파일 등 다양한 마이그레이션 포맷 지원

## 🏗️ 기술 아키텍처

### 시스템 개요

```
┌─────────────────────────────────────┐
│          Migration Files            │
│  (.sql, .ts, prisma schema, etc.)   │
└──────────────┬──────────────────────┘
               │
       ┌───────▼────────┐
       │  Meta Parser   │  (주석/메타데이터 추출)
       └───────┬────────┘
               │
       ┌───────▼────────┐
       │   DAG Builder  │  (graphlib 기반 의존 관계 그래프)
       └───┬───────┬────┘
           │       │
  ┌────────▼─┐  ┌──▼──────────────┐
  │ Conflict │  │  Plan Generator │  (토폴로지 정렬 + 윈도우 필터)
  │ Detector │  └──┬──────────────┘
  └──────────┘     │
               ┌───▼──────────────┐
               │   Ink CLI View   │  (인터랙티브 터미널 UI)
               └──────────────────┘
```

### 기술 스택 선택 이유

- **TypeScript + Node.js**: 기존 JS 생태계 마이그레이션 도구(Prisma, Knex)와 동일한 환경, 파서 작성 용이
- **graphlib**: 검증된 DAG 구현체. 토폴로지 정렬, 사이클 감지 등 그래프 알고리즘 제공
- **Ink (React for CLI)**: 복잡한 터미널 UI(트리 렌더링, 인터랙티브 선택)를 React 컴포넌트 방식으로 작성 가능
- **better-sqlite3**: 마이그레이션 실행 이력 로컬 저장. 별도 DB 서버 불필요

### 주요 기술적 도전

1. **다형 파서 설계**: SQL 주석, TypeScript 파일 export, Prisma 스키마 등 각기 다른 포맷에서 의존 메타데이터를 추출하는 플러그인형 파서 인터페이스 설계
2. **부분 순서(partial order) 시각화**: DAG에서 병렬 실행 가능한 노드 그룹을 "레이어"로 묶어 터미널에서 시각적으로 표현하는 레이아웃 알고리즘
3. **충돌 감지 정확도**: 컬럼명 기반 텍스트 매칭이 아니라 SQL AST 파싱 수준의 충돌 감지 구현

## 🎨 무엇이 흥미로운가?

### 독특한 각도

기존 마이그레이션 도구들은 **실행 엔진**에 집중한다. HarvestPlan은 실행 전 단계인 **계획과 조율**에 집중한다. 농업 달력 메타포는 단순한 이름 붙이기가 아니라, "씨앗 → 성장 → 수확"의 생명주기가 마이그레이션의 "작성 → 검토 → 배포" 생명주기와 구조적으로 동일하다는 통찰에서 나왔다.

### 혁신 포인트

- **배포 조건의 일급 시민화**: 마이그레이션 파일 안에 "언제 실행 가능한가"라는 조건을 코드로 표현
- **정적 분석 레이어**: 실제 DB 없이도 마이그레이션 간 충돌과 의존 관계를 로컬에서 분석
- **농업 메타포의 도메인 크로스오버**: 작물 성장 모델링(선행 조건 충족 후 다음 단계)을 DB 스키마 변화 모델링에 적용

### 학습 기회

- DAG(방향 비순환 그래프) 구현과 토폴로지 정렬 알고리즘
- SQL/TypeScript 파일의 AST 파싱 기초
- Ink를 사용한 복잡한 터미널 UI 컴포넌트 설계
- 플러그인 아키텍처 패턴 (파서 인터페이스)

## 📊 시장과 검증

### 유사 프로젝트

- **Flyway** (flywaydb.org): 강력한 실행 엔진, 하지만 의존 관계 선언과 배포 계획 기능 없음
- **Liquibase** (liquibase.com): changeset 의존 관계 지원하나, 계획 시각화와 윈도우 스케줄링 없음
- **Prisma Migrate** (prisma.io): ORM 연동에 특화, 커스텀 의존 메타데이터 불가
- **Atlas** (atlasgo.io): 스키마 diff 중심, 배포 일정 관리 부재

### 왜 이 공백이 존재하는가

마이그레이션 도구들은 "정확하게 실행"하는 문제를 이미 잘 해결했다. "언제, 어떤 순서로, 무엇과 함께"라는 조율 문제는 팀 슬랙 채널과 스프레드시트로 관리되고 있다. 이 도구는 그 조율을 코드화한다.

- [ ] `harvestplan plan` 출력이 수동 계획 대비 시간 절약 측정 가능

## 🌱 무엇을 배우는가?

### 새로운 기술과 개념

- **DAG와 토폴로지 정렬**: 그래프 이론의 핵심 알고리즘을 실제 문제에 적용하는 깊은 이해
- **AST 파싱 기초**: SQL과 TypeScript 파일을 구조적으로 분석하는 방법론
- **Ink 터미널 UI**: React 패러다임을 CLI에 적용 — 컴포넌트 기반 터미널 렌더링
- **플러그인 아키텍처**: 개방-폐쇄 원칙을 실제 도구에 적용하는 인터페이스 설계

### 성장 기회

- 부분 순서(partial order)와 전체 순서(total order)의 차이를 코드로 체득
- "정적 분석"이 런타임 실행 없이 어떻게 버그를 잡는지 직접 구현
- 복잡한 상태를 가진 CLI 도구에서 UX와 정보 아키텍처 설계

### 도전 구간

SQL AST 파싱 없이 텍스트 매칭만으로는 충돌 감지가 부정확하다. 정확도를 높이려면 간단한 SQL 파서를 직접 구현하거나 `node-sql-parser` 같은 라이브러리를 깊게 파야 한다. 이 지점이 가장 큰 기술적 도전이자 학습 기회다.
