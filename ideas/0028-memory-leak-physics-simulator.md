# 메모리 누수 물리 시뮬레이터

> 메모리 누수를 추상적인 숫자가 아닌 **흘러넘치는 유체**로 시각화하여, 어디서 새는지 직관적으로 찾아낸다.

## 🎯 문제 정의

### 현재 상황
메모리 누수 디버깅은 개발자가 가장 고통스러워하는 작업 중 하나다. Chrome DevTools의 Heap Snapshot은 수십만 개의 오브젝트 목록을 나열하고, Memory Timeline은 단순한 선 그래프만 보여준다. 이 데이터들은 정보는 담고 있지만, "어디서 새고 있는가"를 **직관적으로** 이해하기는 극도로 어렵다.

### 핵심 불편함
- **추상적 숫자의 홍수**: "Detached DOM nodes: 4,231개" — 이게 심각한가? 어디서 생겼나? 한눈에 알 수 없다.
- **정적 스냅샷의 한계**: 시간의 흐름에 따라 메모리가 어떻게 "차오르는지"를 감각적으로 느끼기 어렵다.
- **인과관계 파악 불가**: 특정 사용자 액션이 메모리 증가를 유발한다는 사실을 연결 짓기가 번거롭다.

### 겪는 사람
프론트엔드 성능에 책임을 지는 중급~시니어 개발자. 특히 React 애플리케이션의 컴포넌트 언마운트, 이벤트 리스너 미제거, 클로저 참조 문제를 추적해야 하는 상황.

## 💡 해결책

### 핵심 아이디어
메모리 누수 물리 시뮬레이터는 브라우저의 Performance API와 Chrome DevTools Protocol(CDP)로 수집한 힙 메모리 데이터를 **물리 기반 유체 시뮬레이션**으로 변환한다. 메모리 할당은 용기에 물이 채워지는 것으로, 가비지 컬렉션은 물이 빠지는 것으로, 그리고 누수는 균열에서 새어나오는 흐름으로 표현한다.

각 "용기(컨테이너)"는 JavaScript 컨텍스트(컴포넌트, 모듈, 이벤트 리스너 그룹)에 대응하며, 용기들 사이를 물이 흘러가는 방향이 참조 관계를 나타낸다. GC가 회수하지 못하는 메모리는 용기 밖으로 조금씩 새어나와 바닥에 고인다.

### 작동 방식
1. **연결**: 개발 중인 웹 앱 URL을 입력하면, 시뮬레이터가 Chrome CDP를 통해 힙 프로파일링을 시작한다.
2. **매핑**: 힙 스냅샷의 보유 트리를 파싱하여 각 오브젝트 그룹을 물리 용기로 변환한다. 보유 크기(retained size)가 용기의 최대 수위를 결정한다.
3. **시뮬레이션**: Canvas 위에서 실시간으로 유체가 출렁이고 흘러넘치는 물리 시뮬레이션이 돌아간다. 누수 지점은 균열 이펙트와 함께 강조된다.
4. **인터랙션**: 앱에서 특정 액션(버튼 클릭, 페이지 이동)을 수행하면, 해당 액션 전후의 메모리 변화가 물이 갑자기 쏟아지는 애니메이션으로 표현된다.
5. **진단**: 특정 용기를 클릭하면 실제 오브젝트 타입, 보유 경로, 예상 원인을 텍스트로 제공한다.

### 주요 기능
- **유체 물리 Canvas 렌더링**: 실시간으로 출렁이는 물 표면, 균열에서 새는 흐름을 Canvas 2D로 구현
- **액션-메모리 연결 타임라인**: 사용자가 앱에서 수행한 액션을 기록하고 메모리 변화와 겹쳐 표시
- **누수 심각도 히트맵**: 어떤 컨텍스트가 가장 많이 누수하는지 물의 양(용기 크기)으로 직관적 비교
- **GC 이벤트 시각화**: 가비지 컬렉션 발생 시 물이 빠르게 빠지는 애니메이션으로 표현

## 🏗️ 기술 아키텍처

### 시스템 개요

```
┌─────────────────────────────┐
│   Chrome DevTools Protocol  │
│   (WebSocket via CDP)       │
└──────────────┬──────────────┘
               │ Heap snapshots,
               │ Timeline events
               ▼
┌─────────────────────────────┐
│   파서 & 그래프 빌더         │
│   (힙 보유 트리 → 그래프)   │
└──────────────┬──────────────┘
               │ 정규화된 노드/엣지 데이터
               ▼
┌─────────────────────────────┐
│   물리 시뮬레이션 엔진       │
│   (유체 역학, 입자 시스템)  │
└──────────────┬──────────────┘
               │ 프레임별 렌더 데이터
               ▼
┌─────────────────────────────┐
│   Canvas 2D 렌더러           │
│   (물 표면, 균열, 흐름)     │
└──────────────┬──────────────┘
               │
┌──────────────▼──────────────┐
│   React UI 레이어            │
│   (컨트롤, 진단 패널)       │
└─────────────────────────────┘
```

### 기술 스택 선택 근거
- **Chrome DevTools Protocol**: 브라우저 외부에서 힙 데이터를 프로그래밍적으로 가져오는 유일한 공식 경로. `HeapProfiler.takeHeapSnapshot` API 활용.
- **Canvas 2D API**: 실시간 입자/유체 시뮬레이션에 WebGL 없이도 충분한 성능. 60fps 렌더링 목표.
- **TypeScript**: 힙 스냅샷 데이터 구조가 복잡하므로 타입 안전성 필수.
- **React**: 진단 패널, 타임라인 컨트롤 등 UI 레이어 담당.

### 핵심 기술 도전
1. **힙 스냅샷 파싱 최적화**: V8 힙 스냅샷 포맷은 수십만 노드를 포함할 수 있다. 보유 트리를 의미 있는 "컨텍스트 그룹"으로 클러스터링하는 알고리즘이 필요하다. 접근: 노드 타입과 이름 패턴으로 React 컴포넌트, 이벤트 리스너 등을 구분.
2. **유체 시뮬레이션 성능**: 50개 이상의 용기가 실시간으로 시뮬레이션될 때 Canvas 렌더링 성능 유지. 접근: 입자 수를 동적으로 조절하는 LOD(Level of Detail) 시스템 적용.
3. **CDP 연결 안정성**: CDP WebSocket 연결이 끊겼을 때 재연결 로직과 데이터 손실 없는 스냅샷 동기화.

## 🌱 무엇을 배우게 되나

### 새로운 기술과 개념
- **Chrome DevTools Protocol**: 브라우저 내부를 프로그래밍적으로 제어하는 저수준 API. 이해하면 E2E 테스트, 성능 자동화 등 전 영역에 응용 가능.
- **V8 힙 구조 이해**: JavaScript 엔진이 메모리를 어떻게 관리하는지 내부 구조 수준에서 파악. GC 알고리즘, 보유 경로 개념의 깊은 이해.
- **입자 기반 유체 시뮬레이션**: SPH 알고리즘의 기본 원리. 물리 시뮬레이션 구현 경험은 게임, 인터랙티브 아트 등에 전이 가능.
- **실시간 렌더링 최적화**: requestAnimationFrame 루프에서 데이터 처리와 렌더링을 분리하는 파이프라인 설계.

### 성장 기회
- "브라우저는 블랙박스"라는 인식을 깨고 내부 동작을 직접 관찰하는 자신감
- 물리 시뮬레이션과 데이터 시각화의 경계를 넘나드는 크로스 도메인 사고
- 성능 분석 도구를 사용하는 것에서 만드는 것으로의 전환

### 도전 구간
SPH 유체 시뮬레이션의 안정성 유지 (발산 방지)와 대용량 힙 데이터의 실시간 처리가 이 프로젝트의 가장 큰 기술적 허들이다. 두 영역 모두 깊은 이해 없이는 해결이 어려운 진짜 도전이다.
