# Figma 플러그인 — 생태계 진화 기반 디자인 시스템 건강도 분석기

> 컴포넌트를 생물 종처럼 분석해, 어떤 것이 번성하고 어떤 것이 멸종 위기인지 한눈에 파악하세요.

## 🎯 문제

### 현재 상황

대규모 팀에서 운영하는 Figma 디자인 시스템은 시간이 지날수록 복잡해진다. 컴포넌트가 수백 개를 넘어서면, 어떤 것이 실제로 활발히 사용되고 어떤 것이 방치된 채 기술 부채가 되고 있는지 파악하기가 거의 불가능하다. 기존 Figma의 통계 기능은 단순한 사용 횟수만 보여줄 뿐, 컴포넌트 간 관계나 시스템 전체 건강도를 평가하지 못한다.

### 주요 고통 포인트

- **방치 컴포넌트 문제**: 언제부터인지 아무도 사용하지 않는 컴포넌트가 디자인 시스템을 오염시키지만, 삭제하기도 무서워 그냥 남겨둔다.
- **중복 생성 반복**: 이미 존재하는 컴포넌트를 모르고 비슷한 걸 새로 만드는 일이 반복되어 시스템이 분기한다.
- **진화 방향 파악 불가**: 어떤 컴포넌트가 미래에 핵심이 될지, 어떤 것이 도태될지 예측하는 시각적 수단이 없다.
- **리팩터링 우선순위 없음**: "언젠가 정리해야지"라고 생각하지만, 어디서부터 시작해야 할지 데이터가 없다.

### 누가 겪는가?

Figma 디자인 시스템을 운영하는 디자이너 팀, 특히 컴포넌트 수가 100개를 넘어 복잡도가 폭발한 중대형 팀. 디자인 시스템 담당자(Design System Lead)라면 이 문제를 매달 느끼고 있을 것이다.

## 💡 솔루션

### 핵심 개념

생물학의 생태계 진화 모델을 디자인 시스템 분석에 적용한 Figma 플러그인이다. 각 컴포넌트를 하나의 "생물 종"으로 간주하고, 사용 빈도·변형 수·업데이트 주기·다른 컴포넌트와의 의존 관계를 종합해 "적합도(fitness)" 점수를 계산한다. 그 결과를 생태계 지도로 시각화해, 번성하는 종(활발한 컴포넌트), 멸종 위기종(방치 컴포넌트), 침입종(중복 컴포넌트), 공생 관계(함께 자주 쓰이는 쌍)를 직관적으로 보여준다.

### 작동 방식

1. **스캔**: 플러그인이 Figma 파일 전체를 순회하며 모든 컴포넌트의 사용 인스턴스, 변형, 최종 수정 날짜, 연결 관계를 수집한다.
2. **적합도 계산**: 수집한 데이터를 기반으로 각 컴포넌트에 생태계 적합도 점수를 부여한다. 사용 빈도가 높고 다른 컴포넌트와 잘 연결되며 최근에 업데이트될수록 높은 점수를 받는다.
3. **생태계 지도 렌더링**: 컴포넌트들을 D3.js 버블 차트 + 포스 그래프로 시각화한다. 크기는 사용 빈도, 색상은 건강도(초록→빨강), 연결선은 공생 관계를 나타낸다.
4. **진단 리포트 제공**: "멸종 위기 컴포넌트 5개", "침입종(중복 의심) 3쌍", "이번 달 가장 빠르게 성장한 컴포넌트" 같은 구체적인 액션 아이템을 제시한다.

### 주요 기능

- **생태계 건강도 대시보드**: 전체 시스템을 생태계 지도 형태로 한 화면에 요약
- **멸종 위기종 탐지**: 최근 90일 이상 사용되지 않은 컴포넌트를 자동 감지하고 삭제 후보 목록 제공
- **침입종(중복) 클러스터링**: 이름과 구조가 비슷한 컴포넌트를 묶어 병합 제안
- **공생 관계 맵**: 함께 자주 사용되는 컴포넌트 쌍을 시각화해 조합 가이드라인 도출
- **시계열 진화 재생**: 주별 스냅샷으로 디자인 시스템이 어떻게 진화해왔는지 타임라인 재생

## 🏗️ 기술 아키텍처

### 시스템 개요

```
┌──────────────────────────────┐
│     Figma Plugin UI          │  (D3.js 생태계 지도, 대시보드)
└──────────────┬───────────────┘
               │ postMessage
┌──────────────▼───────────────┐
│     Plugin Sandbox (Main)    │  (TypeScript)
│  - Figma API 스캔            │
│  - 적합도 계산 엔진           │
│  - 관계 그래프 구성           │
└──────────────────────────────┘
```

### 기술 스택 선택 이유

- **Figma Plugin API**: 파일 내 모든 컴포넌트·인스턴스에 접근할 수 있는 유일한 공식 인터페이스
- **TypeScript**: 복잡한 그래프 데이터 구조를 안전하게 다루기 위해
- **D3.js**: 포스 지향 그래프(force-directed graph)와 버블 차트를 조합한 생태계 지도를 렌더링하기에 최적

### 핵심 기술 챌린지

1. **적합도 점수 설계**: 사용 빈도, 관계 밀도, 업데이트 시기를 어떻게 가중 합산할지 결정해야 한다. 단순 선형 합이 아니라 생물학적 적합도 함수(fitness landscape)를 참고해 비선형 모델로 만드는 것이 핵심이다.
2. **Figma API 성능 한계**: 컴포넌트가 수백 개인 대형 파일을 스캔할 때 API 호출 제한에 걸릴 수 있다. 청크 단위 배치 처리와 캐싱 전략이 필요하다.
3. **D3 + Figma Plugin UI 통합**: Figma 플러그인의 iframe 환경에서 D3 포스 그래프를 성능 이슈 없이 렌더링하는 것은 일반 웹 환경과 다른 최적화가 필요하다.

## 🎨 무엇이 흥미로운가?

### 독특한 관점

디자인 시스템 분석 도구는 주로 "커버리지"나 "사용 통계" 관점으로 접근한다. 이 프로젝트는 생물학의 자연선택 이론을 메타포로 삼아 완전히 다른 언어로 같은 문제를 해석한다. "어떤 컴포넌트를 삭제해야 하나요?"가 아니라 "어떤 종이 멸종 위기인가요?"라는 질문으로 전환하는 것 자체가 디자인 시스템 관리에 대한 신선한 인식 변화를 만든다.

### 혁신 포인트

- 생물학 생태계 적합도 모델(fitness landscape)을 UI 컴포넌트 분석에 처음 적용
- 단순 사용 통계를 넘어 컴포넌트 간 공생·경쟁 관계까지 시각화
- 시계열 진화 재생으로 디자인 시스템의 "자연사(natural history)" 개념 도입

### 학습 기회

- Figma Plugin API의 전체 데이터 모델과 컴포넌트 탐색 패턴
- D3.js 포스 지향 그래프 알고리즘 심층 이해
- 생물학적 피트니스 함수의 수학적 구조를 소프트웨어 메트릭에 적용하는 방법
- Figma 플러그인 환경(iframe sandbox)에서의 성능 최적화

## 📊 시장 및 검증

### 유사 프로젝트

- **Figma의 기본 Analytics**: 단순 사용 횟수만 제공. 컴포넌트 간 관계나 건강도 개념 없음.
- **Supernova** (supernova.io): 디자인 토큰 동기화에 집중. 컴포넌트 생태계 분석 없음.
- **Knapsack** (knapsack.cloud): 디자인 시스템 문서화 플랫폼. 분석이 아닌 퍼블리싱 중심.

### 이 공백이 존재하는 이유

생물학적 메타포를 도구 설계에 적용하려면 도메인 간 아이디어 전이가 필요한데, 디자인 툴 생태계에서는 이런 크로스도메인 접근이 거의 시도된 적이 없다. 또한 Figma Plugin API가 충분히 성숙한 것은 비교적 최근의 일이라 본격적인 분석 도구 생태계가 아직 형성 중이다.

### 성공 지표

- [ ] GitHub 스타 200개 돌파 (오픈소스 공개 후 한 달 내)

### 성장 기회

- Figma 플러그인 개발의 독특한 제약(sandbox 환경, 메시지 패싱)을 체험하며 비표준 개발 환경에 대한 적응력 강화

Figma Plugin API의 샌드박스 구조는 일반 웹 개발과 달리 UI 스레드와 메인 스레드가 분리되어 있어, 데이터 스캔과 렌더링을 비동기 메시지 패싱으로 연결해야 한다. 이 아키텍처를 처음 다루는 것이 가장 큰 도전이자 배움의 핵심이다.

## 💭 확장 및 미래 작업

### 자연스러운 확장

- **자동 정리 제안**: "이 컴포넌트를 저 컴포넌트에 병합하세요" 형태의 구체적인 리팩터링 액션 플랜
- **팀 행동 분석**: 특정 디자이너가 어떤 컴포넌트를 주로 만들고 사용하는지 패턴 분석
- **Storybook 연동**: Figma 컴포넌트와 Storybook 스토리의 1:1 매핑 건강도 체크

### 연구 질문

- 생물학적 적합도 함수 이외에 어떤 생태계 메트릭(예: 생물다양성 지수)을 디자인 시스템에 적용할 수 있을까?
- 컴포넌트 간 "기생 관계"(한쪽만 이득을 보는 의존)는 어떻게 정의할 수 있을까?
