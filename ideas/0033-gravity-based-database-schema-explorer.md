# 중력 기반 데이터베이스 스키마 탐험기

> 데이터베이스 테이블들이 중력으로 서로를 끌어당기는 물리 시뮬레이션 — 외래 키 관계를 행성계처럼 탐험하라

## 🎯 문제

### 현재 상황
복잡한 데이터베이스 스키마를 이해하는 것은 고통스럽다. 기존 ERD(Entity-Relationship Diagram) 도구들은 정적인 박스-화살표 다이어그램을 보여주지만, 수십 개의 테이블이 얽혀 있을 때 어떤 테이블이 핵심이고 어떤 테이블이 주변부인지 직관적으로 파악하기 어렵다. 특히 레거시 데이터베이스를 처음 파악하거나, 마이그레이션 계획을 세울 때 관계의 "무게감"을 시각적으로 느끼기 어렵다.

### 고통 지점
- **관계 밀도 파악 불가**: 어떤 테이블이 가장 많은 외래 키 참조를 받는지 ERD에서 한눈에 보기 어렵다
- **중요도 맥락 부재**: 단순한 선 연결은 관계의 강도나 빈도를 표현하지 못한다
- **탐색 피로**: 50개 이상의 테이블이 있는 스키마에서 스크롤하고 확대/축소하다 맥락을 잃는다
- **마이그레이션 영향 파악 어려움**: 특정 테이블을 삭제하거나 수정할 때 파급 효과를 직관적으로 가늠하기 힘들다

### 누가 이 문제를 겪는가?
레거시 시스템을 유지보수하는 백엔드 개발자, 새 팀에 합류해 데이터베이스 구조를 빠르게 파악해야 하는 개발자, 스키마 리팩터링이나 마이그레이션을 계획하는 DBA, 그리고 프론트엔드 개발자 중 API 뒤의 데이터 구조를 이해하고 싶은 사람들.

## 💡 해결책

### 핵심 개념
중력 기반 데이터베이스 스키마 탐험기는 SQL 스키마 DDL을 파싱하여 각 테이블을 물리 시뮬레이션의 질량체(天體)로 변환한다. 외래 키 참조 횟수가 많은 테이블일수록 질량이 크고, 연결된 테이블들은 서로를 중력으로 끌어당긴다. 결과적으로 핵심 테이블들은 자연스럽게 클러스터 중심에 모이고, 주변 테이블들은 궤도를 돌듯 배치된다. 정적인 다이어그램이 아니라 살아 숨 쉬는 물리 시뮬레이션으로 스키마 구조를 탐험한다.

### 작동 방식
1. **스키마 파싱**: SQL DDL 파일 또는 텍스트를 붙여넣으면 테이블, 컬럼, 외래 키 관계를 추출한다
2. **질량 계산**: 각 테이블의 질량은 인바운드 외래 키 수 × 가중치 + 컬럼 수로 결정된다 — 참조 많이 받는 테이블 = 블랙홀
3. **중력 시뮬레이션 시작**: Matter.js 기반 물리 엔진이 실행되며 테이블들이 자연스러운 균형 위치를 찾아간다
4. **인터랙티브 탐색**: 특정 테이블을 클릭하면 직접 연결된 테이블만 하이라이트되고 나머지는 희미해진다
5. **영향 분석 모드**: "이 테이블을 제거하면?" 버튼을 누르면 해당 테이블이 시뮬레이션에서 사라지며 나머지 테이블들의 재배치 애니메이션을 보여준다

### 주요 기능
- **중력 클러스터링**: 외래 키로 연결된 테이블들이 자동으로 뭉쳐 도메인 경계를 시각화
- **질량 시각화**: 테이블 크기가 중요도(참조 횟수)에 비례해 표시되어 "핵심 테이블" 즉시 파악
- **파급 효과 시뮬레이터**: 테이블 삭제/분리 시 나머지 구조가 어떻게 재편되는지 물리적으로 확인
- **도메인 자동 감지**: 중력 클러스터를 분석해 `user_*`, `order_*` 등 자연스러운 도메인 경계를 컬러로 표시
- **SQL DDL 직접 붙여넣기**: 파일 없이 텍스트만으로 즉시 시각화

## 🏗️ 기술 아키텍처

### 시스템 개요

```
┌─────────────────────────────┐
│     SQL DDL 입력 (텍스트)     │
└──────────────┬──────────────┘
               │
┌──────────────▼──────────────┐
│        스키마 파서            │  (node-sql-parser 또는 직접 구현)
│  테이블, 컬럼, FK 관계 추출    │
└──────────────┬──────────────┘
               │
┌──────────────▼──────────────┐
│       질량/인력 계산기         │  (TypeScript 순수 로직)
│  질량 = FK_in * 3 + col * 0.5 │
│  인력 = FK 존재 시 스프링 연결  │
└──────────────┬──────────────┘
               │
       ┌───────┴───────┐
       │               │
┌──────▼──────┐  ┌──────▼──────┐
│ Matter.js   │  │   D3.js     │
│ 물리 시뮬레이션│  │ SVG 렌더링   │
│ 중력/충돌/질량│  │ 레이블/툴팁  │
└──────┬──────┘  └──────┬──────┘
       └───────┬─────────┘
               │
┌──────────────▼──────────────┐
│       React UI 레이어        │
│  컨트롤 패널, 테이블 상세뷰    │
└─────────────────────────────┘
```

### 기술 스택 선택 이유
- **Matter.js**: 브라우저용 2D 물리 엔진 중 가장 성숙하고 TypeScript 지원이 좋음. 중력, 스프링, 충돌 등 모든 필요 기능 내장
- **D3.js**: Matter.js가 계산한 물리 좌표를 SVG로 렌더링. 포스 레이아웃 대신 Matter.js를 물리 엔진으로 사용하고 D3는 순수 렌더링만 담당하는 독특한 조합
- **node-sql-parser**: SQL DDL 파싱 라이브러리. PostgreSQL, MySQL 문법 모두 지원
- **TypeScript + Vite**: 빠른 개발 환경과 타입 안전성

### 핵심 기술 과제
1. **Matter.js와 D3 렌더링 동기화**: Matter.js의 물리 루프(60fps)와 D3 SVG 업데이트를 효율적으로 연결하는 것이 핵심. requestAnimationFrame으로 Matter.js 상태를 D3에 반영하는 파이프라인 구축 필요
2. **대형 스키마 성능**: 테이블 100개 이상일 때 물리 시뮬레이션이 느려질 수 있음. LOD(Level of Detail) — 멀리 있는 테이블은 단순화된 물체로 처리 — 도입 고려
3. **중력 균형점 찾기**: 모든 테이블이 한 점으로 뭉치지 않도록 척력(repulsion force)과 인력의 균형 파라미터 튜닝

## 🎨 무엇이 흥미로운가?

### 독창적인 각도
데이터베이스 시각화 도구는 모두 "박스와 화살표" 패러다임에 갇혀 있다. 이 도구는 물리학의 중력 모델을 데이터 관계에 적용한 첫 시도다. 외래 키를 중력으로 해석하면 — 많이 참조받는 테이블은 무겁고, 덜 중요한 테이블은 가볍게 궤도를 도는 — 데이터베이스 구조가 직관적인 공간적 의미를 갖게 된다.

### 혁신 포인트
- **물리학 은유의 데이터베이스 적용**: 천체물리학의 중력 클러스터링 개념을 ER 다이어그램에 접목
- **Matter.js + D3 하이브리드**: 물리 시뮬레이션 엔진을 데이터 시각화 렌더러와 조합하는 비전형적 아키텍처
- **정적 → 동적 구조 이해**: 스키마를 "보는" 것이 아니라 "느끼는" 경험
- **파급 효과 물리 시뮬레이션**: 마이그레이션 영향을 수식이 아닌 물리 애니메이션으로 예측

### 학습 기회
- Matter.js 물리 엔진 깊이 이해 (강체, 제약, 스프링)
- D3 커스텀 렌더링 루프 구현
- SQL 파싱 및 AST 처리
- 물리 시뮬레이션 파라미터 튜닝 경험

## 📊 시장 및 검증

### 유사 프로젝트
- **DBDiagram.io** (dbdiagram.io): 정적 ERD 도구. 물리 시뮬레이션 없음, 관계 중요도 시각화 없음
- **SchemaSpy** (github.com/schemaspy/schemaspy): HTML 리포트 생성기. 인터랙티브하지 않음
- **DBeaver ERD**: IDE 내장 다이어그램. 정적이며 수동 배치 필요
- **D3 Force Layout** (개인 구현들): D3 포스 레이아웃을 쓰는 시도는 있지만 진정한 물리 시뮬레이션(질량, 중력 상수, 스프링 탄성)을 구현한 것은 없음

### 왜 이 공백이 존재하는가?
데이터베이스 도구는 전통적으로 "정확성"을 우선시하고 "느낌"을 희생한다. ERD의 공간적 배치는 완전히 자의적이다. 물리 시뮬레이션을 적용하면 배치에 의미가 생기지만, 물리 엔진과 데이터 시각화를 조합하는 것이 기술적으로 익숙하지 않은 영역이라 시도가 드물다.

## 🌱 무엇을 배울 것인가?

### 새로운 기술과 개념
- **Matter.js 물리 엔진**: 강체(rigid body), 제약(constraint), 복합체(composite), 스프링 동역학 — 게임 개발에서 주로 쓰이는 기술을 데이터 시각화에 적용하는 경험
- **물리 시뮬레이션 + D3 렌더링 파이프라인**: 두 독립적인 루프(물리 루프, 렌더 루프)를 효율적으로 연결하는 아키텍처 패턴
- **SQL 파싱**: 컴파일러 이론의 실용적 적용 — DDL 문법을 AST로 파싱하고 관계 그래프로 변환
- **파라미터 기반 물리 튜닝**: 중력 상수, 척력 강도, 댐핑 계수를 조정하며 창발적 동작을 이해하는 경험

### 성장 기회
- 물리 시뮬레이션 파라미터를 조정하며 "창발적 구조화"가 어떻게 동작하는지 직접 체험
- 두 개의 독립적 렌더링 시스템을 하나의 인터페이스로 통합하는 아키텍처 설계 능력
- 성능 병목(물리 계산 vs. DOM 업데이트)을 프로파일링하고 최적화하는 경험

### 도전 구간
물리 시뮬레이션 파라미터 튜닝이 핵심 과제다. 중력이 너무 강하면 모든 것이 뭉치고, 너무 약하면 흩어진다. 이 균형을 잡는 과정에서 물리 엔진의 내부 동작을 깊이 이해하게 된다.
